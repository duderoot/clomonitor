== Overview

The `clomonitor-registrar` is a Rust backend component that registers projects from various data sources into the CLOMonitor PostgreSQL database. It supports two primary data sources:

1. **Foundation YAML files**: Traditional YAML files served over HTTP (CNCF, LF AI & DATA, CDF)
2. **Dependency-Track instances**: Fetches components from DT API and maps them to CLOMonitor projects

=== Sequence Diagram: YAML Foundation Processing

title Runtime View - YAML Foundation Processing

actor Admin
participant "Main" as main
participant "Registrar" as reg

participant "YAML File" as yaml

Admin -> main: Run with config

reg -> db: foundations()
activate db
db -> pg: SELECT foundation_id, \ndata_source_type, data_source_config
pg --> db: Foundation rows
db --> reg: Vec<Foundation>
deactivate db

loop For each foundation (concurrent, configurable limit)
    reg -> reg: Timeout(300s)

    alt data_source == YamlUrl
        reg -> http: GET data_url
        activate http
        http -> yaml: HTTPS GET
        yaml --> http: YAML content
        http --> reg: Response body
        deactivate http

        reg -> reg: Parse YAML to Vec<Project>
        reg -> reg: Filter excluded repos
        reg -> reg: Calculate digests

        reg -> db: foundation_projects(foundation_id)
        activate db
        db -> pg: SELECT project_name, digest
        pg --> db: Existing projects
        db --> reg: HashMap<name, digest>
        deactivate db

            alt New or digest changed
                reg -> db: register_project(foundation_id, project)
                activate db
                db -> pg: SELECT register_project($1::text, $2::jsonb)
                pg --> db: Success
                db --> reg: Ok()
                deactivate db
            end
        end

        loop For each existing project not in YAML
            reg -> db: unregister_project(foundation_id, name)
            activate db
            db -> pg: SELECT unregister_project($1::text, $2::text)
            pg --> db: Success
            db --> reg: Ok()
            deactivate db
        end
    end
end

reg --> main: Ok()
deactivate reg
main --> Admin: Exit
deactivate main

=== Sequence Diagram: Dependency-Track Foundation Processing

[plantuml, runtime-dt, svg]
....
@startuml
title Runtime View - Dependency-Track Foundation Processing

participant "Registrar" as reg
participant "DT Client" as dt_client
participant "DT Mapper" as mapper
participant "Registry Router" as router
participant "npm Client" as npm
participant "DB Layer" as db
participant "DT API" as dt_api
participant "npm Registry" as npm_api
database "PostgreSQL" as pg

reg -> dt_client: get_projects()
activate dt_client

loop Pagination (page 1, 2, 3...)
    dt_client -> dt_api: GET /api/v1/project\n?pageNumber=N&pageSize=100\nX-Api-Key: xxx
    dt_api --> dt_client: [DtProject] + X-Total-Count

    alt Status 429 (Rate Limited)
        dt_client -> dt_client: Sleep with exponential backoff
        dt_client -> dt_api: Retry GET request
    end
end

dt_client --> reg: Vec<DtProject>
deactivate dt_client

loop For each DT project
    reg -> dt_client: get_project_components(project_uuid)
    activate dt_client

    loop Pagination
        dt_client -> dt_api: GET /api/v1/component/project/{uuid}\n?pageNumber=N&pageSize=100
        dt_api --> dt_client: [DtComponent] + X-Total-Count
    end

    dt_client --> reg: Vec<DtComponent>
    deactivate dt_client

    loop For each component
        reg -> mapper: should_process_component(component)
        activate mapper
        mapper --> reg: bool (LIBRARY or FRAMEWORK)
        deactivate mapper

        alt Should process
            reg -> mapper: extract_repository_url_with_lookup(\ncomponent, registry_router, db)
            activate mapper

            alt Has externalReferences[vcs]
                mapper --> reg: Some(repo_url)
            else purl is GitHub/GitLab
                mapper -> mapper: extract_url_from_purl()
                mapper --> reg: Some(repo_url)
            else Check database cache
                mapper -> db: get_component_mapping(purl)
                activate db
                db -> pg: SELECT repo_url FROM component_mapping\nWHERE purl = $1
                pg --> db: repo_url or NULL
                db --> mapper: Option<String>
                deactivate db

                alt Cache hit
                    mapper --> reg: Some(cached_url)
                else Cache miss & registry enabled
                    mapper -> router: lookup_repository_url(purl)
                    activate router

                    router -> router: Parse purl type

                    alt purl type = npm
                        router -> npm: get_repository_url(package_name)
                        activate npm
                        npm -> npm_api: GET /{package_name}
                        npm_api --> npm: Package metadata JSON
                        npm -> npm: Extract repository.url
                        npm --> router: Option<String>
                        deactivate npm
                    end

                    router --> mapper: Option<String>
                    deactivate router

                    alt Lookup successful
                        mapper -> db: save_component_mapping(\npurl, repo_url, "npm", confidence)
                        activate db
                        db -> pg: INSERT INTO component_mapping
                        db --> mapper: Ok()
                        deactivate db
                        mapper --> reg: Some(repo_url)
                    else Lookup failed
                        mapper -> db: save_unmapped_component(\nfoundation_id, component)
                        activate db
                        db -> pg: INSERT INTO unmapped_component
                        db --> mapper: Ok()
                        deactivate db
                        mapper --> reg: None
                    end
                end
            end
            deactivate mapper

            alt repo_url found
                reg -> mapper: convert_component_to_project(\ncomponent, dt_project_name)
                activate mapper
                mapper -> mapper: Format names (name, display_name)
                mapper -> mapper: Create Repository struct
                mapper -> mapper: Create Project struct
                mapper -> mapper: Calculate digest
                mapper --> reg: Result<Project>
                deactivate mapper

                reg -> reg: Add to all_projects_to_register
            end
        end
    end
end

reg -> db: foundation_projects(foundation_id)
activate db
db -> pg: SELECT project_name, digest
pg --> db: Existing projects
db --> reg: HashMap<name, digest>
deactivate db

loop For each project to register
    alt New or changed
        reg -> db: register_project(foundation_id, project)
        activate db
        db -> pg: SELECT register_project($1::text, $2::jsonb)
        pg --> db: Success
        db --> reg: Ok()
        deactivate db
    end
end

loop For each removed project
    reg -> db: unregister_project(foundation_id, name)
    activate db
    db -> pg: SELECT unregister_project($1::text, $2::text)
    pg --> db: Success
    db --> reg: Ok()
    deactivate db
end

=== Activity Diagram: Component URL Resolution Strategy

[plantuml, activity-url-resolution, svg]
....
@startuml
title Activity Diagram - Component Repository URL Resolution

start

:Receive DtComponent;

if (Has externalReferences?) then (yes)
  :Iterate through externalReferences;
  if (Found type="vcs"?) then (yes)
    :Extract URL from vcs reference;
    :Normalize URL (remove .git, trailing slash);
    stop
  endif
endif

:Check purl field;

if (purl starts with "pkg:github/"?) then (yes)
  :Extract "pkg:github/owner/repo@version";
  :Build "https://github.com/owner/repo";
  stop
elseif (purl starts with "pkg:gitlab/"?) then (yes)
  :Extract "pkg:gitlab/owner/repo@version";
  :Build "https://gitlab.com/owner/repo";
  stop
endif

:Query database cache;
:SELECT repo_url FROM component_mapping\nWHERE purl = ?;

if (Cache hit?) then (yes)
  :Return cached repo_url;
  stop
endif

if (Registry API lookup enabled?) then (yes)
  :Parse purl type (npm, maven, pypi);

  if (Type is npm?) then (yes)
    :Call npm registry API;
    :Parse repository.url from package.json;
  elseif (Type is maven?) then (yes)
    :Call Maven Central API;
    :Parse SCM URL from POM;
  elseif (Type is pypi?) then (yes)
    :Call PyPI JSON API;
    :Parse project_urls or home_page;
  else (unknown)
    :No lookup available;
  endif

  if (Lookup successful?) then (yes)
    :Save to component_mapping table;
    :Return resolved URL;
    stop
  else (failed)
    :Save to unmapped_component table;
    :Log: Component skipped, no URL found;
    stop
  endif
else (disabled)
  :Save to unmapped_component table;
  :Log: Component skipped, no URL found;
  stop
endif

== Design Decisions

=== ADR-001: Support Multiple Data Sources

*Status*: Accepted

*Context*: Originally supported only YAML files. Need to support Dependency-Track instances.

*Decision*: Use enum-based polymorphism with `DataSource` enum to support both YAML and DT sources.

*Consequences*:
- Foundation table schema extended with `data_source_type` and `data_source_config`
- Backward compatible with existing YAML foundations
- Easy to add new source types in the future

=== ADR-002: Component URL Resolution Strategy

*Status*: Accepted

*Context*: DT components may not have repository URLs. Need fallback strategy.

*Decision*: Multi-tier resolution:
1. externalReferences (vcs type)
2. Extract from purl (GitHub/GitLab)
3. Database cache lookup
4. Registry API lookup (npm, Maven, PyPI)
5. Save as unmapped

*Consequences*:
- Maximizes successful mappings
- Caching reduces API calls
- Unmapped components tracked for visibility

=== ADR-003: Registry API Lookup Integration

*Status*: Accepted

*Context*: Many components only have purl, not repository URLs.

*Decision*: Integrate with package registry APIs (npm, Maven Central, PyPI) to resolve repository URLs.

*Consequences*:
- Increases mapping success rate
- Adds external dependencies
- Requires rate limiting and caching
- Configurable to disable if needed

== Performance Considerations

=== Concurrency Model

- Uses Tokio async runtime
- Configurable concurrency limit for foundation processing
- 300-second timeout per foundation
- Concurrent HTTP requests with connection pooling

=== Scalability

- Database connection pooling (deadpool-postgres)
- Component mapping cache reduces duplicate lookups
- Registry API responses cached in database
- Pagination for large DT projects (100 items/page)

=== Error Handling

- Retry logic for rate-limited requests (HTTP 429)
- Exponential backoff for retries
- Continue processing on individual component failures
- Comprehensive logging for debugging

== Security Considerations

- TLS required for database connections
- API keys stored in database (foundation config)
- No plaintext credential storage in config files
- Input validation on external data (YAML, JSON)
- SQL injection prevention (parameterized queries)

== Monitoring and Observability

=== Logging

- Structured logging with `tracing` crate
- Configurable log levels via `RUST_LOG`
- JSON format support for log aggregation
- Per-foundation instrumentation

=== Metrics

Key metrics to track:
- Foundation processing duration
- Number of projects registered/unregistered
- Component mapping success/failure rates
- Registry API call counts
- Database operation latencies

== Future Enhancements

=== Potential Improvements

1. **GraphQL API Support**: Add support for GraphQL-based data sources
2. **Webhook Integration**: Support real-time updates via webhooks
3. **Multi-region DT**: Support multiple DT instances per foundation
4. **Component Deduplication**: Detect and merge duplicate components across projects
5. **Machine Learning**: Use ML to improve URL extraction confidence scores
6. **Incremental Updates**: Support incremental syncs instead of full scans
